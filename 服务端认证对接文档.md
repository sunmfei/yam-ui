# 服务端认证对接文档

## 1. 对接目标

认证服务用于给前端 SPA 提供完整的 OAuth2 + OIDC 能力，并通过后端适配接口解决：

- 前端无法安全持有 `client_secret`
- 浏览器跨域访问认证服务
- 授权码换 token、刷新 token、用户信息读取、会话登出

当前服务约定：

- 服务：`service-auth-security`
- 服务端口：`48081`
- `context-path=/auth`

最终外部访问根路径：

- `http://localhost:48081/auth`

---

## 2. 服务端基础配置

### application.yml

关键配置：

```yml
server:
  base-url: http://localhost:48081/auth
  port: 48081
  servlet:
    context-path: /auth
```

说明：

- `context-path=/auth` 决定了所有对外路径都带 `/auth`
- `server.base-url` 必须与实际对外访问根路径一致
- 不要再配置成不带 `/auth` 的老地址

---

## 3. OAuth2 客户端配置

当前 Web 客户端：

```yml
oauth2:
  clients:
    - client-id: web-app
      client-secret: "{noop}123456"
      authentication-method: client_secret_basic
      authorization-grant-types:
        - authorization_code
        - refresh_token
        - client_credentials
      redirect-uris:
        - http://localhost:53001/callback
      require-authorization-consent: false
      require-proof-key: false
```

关键点：

- `web-app` 使用 `client_secret_basic`
- 前端不能直接换 token
- 必须通过服务端适配接口中转

---

## 4. 标准 OAuth2 / OIDC 端点

### 登录

```http
POST /auth/login
```

### 授权

```http
GET /auth/oauth2/authorize
```

### Token

```http
POST /auth/oauth2/token
```

### 用户信息

```http
GET /auth/userinfo
```

### OIDC 发现

```http
GET /auth/.well-known/openid-configuration
```

---

## 5. 提供给前端的适配接口

前端不直接使用 `/oauth2/token`，而通过认证服务自定义的适配接口：

### 5.1 callback 换 token

```http
POST /auth/callback
```

请求体：

```json
{
  "code": "授权码",
  "redirectUri": "http://localhost:53001/callback"
}
```

作用：

- 后端用 `web-app` 的 `client_secret` 调用 `/auth/oauth2/token`
- 返回：
  - `access_token`
  - `refresh_token`
  - `id_token`
  - `expires_at`
  - `userInfo`

### 5.2 refresh

```http
POST /auth/refresh
```

请求体：

```json
{
  "refreshToken": "xxx"
}
```

作用：

- 调用 `/auth/oauth2/token`
- `grant_type=refresh_token`
- 返回新的 token 集合和用户信息

### 5.3 当前用户

```http
GET /auth/me
Authorization: Bearer {access_token}
```

作用：

- 将 JWT Claims 标准化为前端可直接使用的用户结构

### 5.4 会话登出

```http
POST /auth/session/logout
```

请求体：

```json
{
  "accessToken": "xxx",
  "refreshToken": "xxx"
}
```

作用：

- 清理 Spring Security 会话
- 删除授权记录
- 返回 JSON 成功结果
- 不做页面重定向

---

## 6. 服务端内部职责划分

### `SecurityConfig`

负责：

- OAuth2 / OIDC 安全过滤链
- JWT 资源服务器能力
- 认证入口
- 对前端适配接口关闭 CSRF
- 会话和 OAuth 端点的安全策略

### `FrontendAuthController`

负责：

- `/callback`
- `/refresh`
- `/me`
- `/session/logout`

它是前端 SPA 对接认证服务的核心适配层。

### `ServerProperties`

负责：

- 服务根地址
- authorization/token/userinfo/jwks 等端点拼接

必须和 `context-path=/auth` 保持一致。

---

## 7. 服务端实现约束

### 7.1 所有内部拼接地址都必须带 `/auth`

例如：

- `http://localhost:48081/auth/oauth2/token`
- `http://localhost:48081/auth/api/auth/me`

不要再使用：

- `http://localhost:48081/oauth2/token`

否则在启用 `context-path=/auth` 时会直接 404。

### 7.2 统一返回 `CommonResult`

当前服务端对外推荐统一返回：

- `site.mfei.core.model.response.CommonResult`

成功示例：

```json
{
  "code": "200",
  "message": "操作成功",
  "data": {},
  "timestamp": 1777115334098,
  "errorDetails": null,
  "traceId": null
}
```

不要在同一个服务里混用：

- `site.mfei.common.model.response.Result`
- `site.mfei.core.model.response.CommonResult`

### 7.3 退出登录不要重定向

给前端 SPA 提供的登出接口必须返回 JSON，不要再 302 到：

```txt
/login?logout
```

否则浏览器 `fetch` 会因为跨域跳转报错。

因此推荐前端只调用：

```http
POST /auth/session/logout
```

不要调用默认：

```http
POST /auth/logout
```

---

## 8. 跨域要求

认证服务至少需要允许：

```yml
cors:
  allowed-origins:
    - http://localhost:53001
  allow-credentials: true
  allowed-methods:
    - GET
    - POST
    - PUT
    - DELETE
    - OPTIONS
  allowed-headers:
    - "*"
```

但开发环境下更推荐由前端使用同源代理 `/auth/*`，这样浏览器不直接跨域打认证服务。

---

## 9. 调试排查指南

### 9.1 登录成功但授权又跳回登录页

优先检查：

- Cookie `JSESSIONID` 的 `Path`
- 前端代理前缀是否与 `context-path` 一致

当前正确组合：

- 后端 `context-path=/auth`
- 前端代理前缀 `/auth`
- Cookie `Path=/auth`

### 9.2 callback 404

通常说明后端内部还在访问旧地址：

- `/oauth2/token`

而不是：

- `/auth/oauth2/token`

### 9.3 前端提示“操作成功”却报错

说明前端把：

```json
"code": "200"
```

误判成失败。  
前端必须兼容字符串 `"200"`。

### 9.4 logout 触发 CORS 错误

说明前端还在调默认：

- `/auth/logout`

并且后端在 302 跳页面。  
应改成：

- `/auth/session/logout`

---

## 10. 联调顺序建议

推荐服务端联调按以下顺序逐步验证：

1. `POST /auth/login`
2. `GET /auth/oauth2/authorize`
3. `POST /auth/callback`
4. `GET /auth/me`
5. `POST /auth/refresh`
6. `POST /auth/session/logout`

---

## 11. 总结

服务端对接的核心原则只有三条：

1. 所有真实地址都必须与 `context-path=/auth` 一致
2. 前端不直接调用 `/oauth2/token`，统一走适配接口
3. 前端专用登出必须返回 JSON，不要做页面重定向

最终推荐前端对接入口为：

- `/auth/login`
- `/auth/oauth2/authorize`
- `/auth/callback`
- `/auth/refresh`
- `/auth/me`
- `/auth/session/logout`
