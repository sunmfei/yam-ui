# 前端认证对接文档

## 1. 对接目标

当前项目采用：

- 前端自建登录页
- 认证服务使用 OAuth2 + OIDC
- 服务端 `context-path=/auth`
- 前端开发环境通过 Vite 代理把同源 `/auth/*` 转发到认证服务

前端不直接调用 `/oauth2/token`，而是通过认证服务提供的适配接口完成：

- 授权码换 Token
- Refresh Token 刷新
- 当前用户信息读取
- 后端会话登出

---

## 2. 环境约定

### 前端地址

- 开发环境：`http://localhost:53001`

### 认证服务地址

- 实际服务根地址：`http://localhost:48081/auth`

### 前端代理前缀

前端统一使用：

- `/auth/*`

由 Vite 转发到：

- `http://localhost:48081/auth/*`

这样可以解决：

- Cookie `Path=/auth` 与浏览器请求路径不一致的问题
- 浏览器直接跨域访问认证服务的问题

---

## 3. 前端环境变量

`.env.development`

```env
VITE_AUTH_ISSUER=http://localhost:48081/auth
VITE_AUTH_BROWSER_BASE_URL=/auth
VITE_AUTH_CLIENT_ID=web-app
VITE_AUTH_SCOPES=openid profile email read write
VITE_AUTH_CALLBACK_PATH=/callback
```

关键说明：

- `VITE_AUTH_ISSUER` 是认证服务真实地址
- `VITE_AUTH_BROWSER_BASE_URL` 是前端浏览器访问时使用的同源代理前缀

---

## 4. 前端代理配置

`vite.config.ts`

```ts
proxy: {
  '/auth': {
    target: env.VITE_AUTH_ISSUER,
    changeOrigin: true,
    secure: false,
    ws: false,
    rewrite: (path) => path.replace(/^\/auth/, ''),
  },
}
```

注意：

- 认证服务已经带 `context-path=/auth`
- 所以 `target` 应配置为 `http://localhost:48081/auth`
- 前端请求 `/auth/login`
- 转发后实际访问 `http://localhost:48081/auth/login`

---

## 5. 前端调用链路

### 5.1 登录

前端登录页提交：

```http
POST /auth/login
Content-Type: application/x-www-form-urlencoded
```

请求体：

```txt
username=admin&password=123456
```

返回成功后，浏览器会拿到：

- `JSESSIONID`

该 Cookie 由认证服务设置，`Path=/auth`。

---

### 5.2 发起授权

登录成功后，前端跳转：

```http
GET /auth/oauth2/authorize
```

典型参数：

```txt
response_type=code
client_id=web-app
redirect_uri=http://localhost:53001/callback
scope=openid profile email read write
state=随机字符串
```

认证服务成功后会回跳：

```txt
http://localhost:53001/callback?code=xxx&state=xxx
```

---

### 5.3 回调换 Token

前端回调页从 URL 取出 `code` 和 `state` 后，调用：

```http
POST /auth/callback
Content-Type: application/json
```

请求体：

```json
{
  "code": "授权码",
  "redirectUri": "http://localhost:53001/callback"
}
```

成功响应示例：

```json
{
  "code": "200",
  "message": "操作成功",
  "data": {
    "tokens": {
      "access_token": "...",
      "refresh_token": "...",
      "id_token": "...",
      "token_type": "Bearer",
      "expires_in": 3600,
      "expires_at": 1777101827384,
      "scope": "read openid profile write email"
    },
    "userInfo": {
      "sub": "admin",
      "name": "超级管理员",
      "preferred_username": "超级管理员",
      "roles": [],
      "permissions": []
    }
  }
}
```

前端保存到：

- `systemCache`
- `sys_token`
- `sys_refreshToken`
- `sys_idToken`
- `sys_tokenExpiresAt`
- `sys_userInfo`

---

### 5.4 获取当前用户

```http
GET /auth/me
Authorization: Bearer {access_token}
```

用于：

- 回调后补齐用户信息
- 页面刷新后恢复用户态

---

### 5.5 刷新 Token

```http
POST /auth/refresh
Content-Type: application/json
```

请求体：

```json
{
  "refreshToken": "xxx"
}
```

响应结构与 `/auth/callback` 一致，也会返回：

- `tokens`
- `userInfo`

---

### 5.6 登出

```http
POST /auth/session/logout
Content-Type: application/json
```

请求体：

```json
{
  "accessToken": "xxx",
  "refreshToken": "xxx"
}
```

作用：

- 清理后端会话
- 删除后端授权记录
- 返回 JSON 成功响应

前端同时需要：

- 清空 `systemCache`
- 清空用户 store
- 跳转 `/login`

---

## 6. 前端代码职责划分

### `src/config/auth.ts`

负责：

- 认证基础地址
- 浏览器访问前缀
- 各认证端点定义

### `src/api/auth.ts`

负责：

- 登录请求
- callback 换 token
- refresh
- me
- session logout

### `src/stores/business/user.ts`

负责：

- 保存认证状态
- 从 `systemCache` 恢复
- 统一处理登录、回调、刷新、登出

### `src/views/auth/*`

负责：

- 登录页展示
- 回调页处理

## 7. 前端注意事项

### 7.1 不要直接调用 `/oauth2/token`

原因：

- `web-app` 是 `client_secret_basic`
- 前端不能暴露 `client_secret`

因此必须通过后端适配接口：

- `/auth/callback`
- `/auth/refresh`

### 7.2 成功码判断要兼容字符串

当前后端返回 `CommonResult`，成功码是：

```json
"code": "200"
```

不是数字 `200`。

前端判断时要兼容：

- `"200"`
- `200`

### 7.3 认证代理前缀必须与 Cookie Path 匹配

当前认证服务 Cookie：

- `Path=/auth`

所以前端也必须使用：

- `/auth/*`

不要再使用：

- `/auth-server/*`

否则浏览器不会自动带上认证 Cookie。

---

## 8. 前端联调检查清单

### 登录前检查

- 认证服务已启动
- 前端 dev 服务已启动
- `.env.development` 已正确配置
- Vite 已重启

### 登录成功标志

- `/auth/login` 返回 200
- `/auth/oauth2/authorize` 成功回跳并带 `code`
- `/auth/callback` 返回 token 集合
- `/auth/me` 返回当前用户信息

### 登出成功标志

- `/auth/session/logout` 返回 200
- 前端缓存被清空
- 再访问受保护页面会回到登录页

---

## 9. 推荐联调顺序

1. 先测 `/auth/login`
2. 再测 `/auth/oauth2/authorize`
3. 再测 `/auth/callback`
4. 再测 `/auth/me`
5. 最后测 `/auth/session/logout`

---

## 10. 总结

前端只需要记住一套统一规则：

- 登录：`/auth/login`
- 授权：`/auth/oauth2/authorize`
- 回调换 token：`/auth/callback`
- 刷新：`/auth/refresh`
- 当前用户：`/auth/me`
- 登出：`/auth/session/logout`

并且：

- 所有认证状态缓存在 `systemCache`
- 前端不直接碰 `client_secret`
- 所有 OAuth2 token 交换统一由后端适配接口完成
